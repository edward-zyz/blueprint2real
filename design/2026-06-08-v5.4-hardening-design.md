## 1. 底盘包(`bootstrap/workflow/`)

> 单向依赖最底层:契约包与模板包都引用这里的脚本行为。先实施、可独立验证。

### 1.1 lint-redlines 修根 + redlineCommands(O14)

**改 `bootstrap/workflow/scripts/lint-redlines.mjs`:**
- root 解析改为从 `process.cwd()`(`npm run` 在 devRoot/workflow 下执行)上溯定位最近的 `workflow.config.mjs`,以其所在 devRoot 为根。**删除任何 fallback 到脚本自身 `__dirname` 所在仓的逻辑**——这是「root 错解析到 skill 模板仓」的根因。
- 定位不到 devRoot → `exit 1` 显式报错「未找到 workflow.config.mjs,lint:redlines 无法确定项目根」,绝不静默指向 skill 仓返回 0。
- 读 `config.redlineCommands`(可选 `string[]`,projectRoot 相对命令)。对每条:在 projectRoot 下执行,退出码非 0 = 红线命中,聚合输出失败命令名 + 尾部 N 行输出,整体 `exit 1`。
- 内建 `RULES` 数组逐条仍照跑(保留现有 grep 类红线能力)。
- **占位态判定**:`RULES` 为空 **且** `redlineCommands` 未配置 → 输出 `[lint-redlines] WARN · 红线未配置,本 gate 无保护力 · root=<devRoot>`,`exit 0`(不阻断,但消灭「OK · 0 规则」假绿观感)。

**改 `workflow.config.mjs` 模板(`bootstrap/workflow/...config.mjs.tmpl`):**
- 增注释占位:`// redlineCommands: ['npm run lint:packages-architecture'],  // 可选,projectRoot 下的真架构/分层/安全 lint,Stage 4 主线亲跑`

**配套 SKILL.md(契约包 2.2 引用)**:Stage 4 主线亲跑 `lint:redlines`,其内部已串起 `redlineCommands`。

### 1.2 validate-state 增 D4 + D5(O13 配套,治孤儿目录/缺 receipt)

**改 `bootstrap/workflow/scripts/validate-state.mjs`:**
- **D4(error)**:遍历 queue.md 中 Status=Done 的工单,按其 level 校验 `work/<slug>/<receiptsDir>/` 下应有的 stage receipt 文件存在:
  - L0 → `5-handoff.json`
  - L1 → `2a-spec.json`(或合并 receipt)、`3-impl.json`、`5-handoff.json`
  - L2/L3 → 全套(`0-triage` 在 work 或 queue 旁、`2a-spec`/`2b-plan`/`2c-review`/`3-impl`/`4-arch`(L3)/`5-handoff`)
  - 缺失任一 → error。**祖父豁免**:工单 ID 在 `state/receipt-grandfather.json.ids[]` 内则跳过 D4(存量历史 Done 工单不被新规则追溯)。
- **D5(warn)**:遍历 `work/` 下目录名,若无法被 queue.md 中任一工单经 `workItemSlug({workId,title})` 命中:
  - 但按 workId 前缀(目录名 `<prefix>-<num>...` 或 `<prefix>-<ts>...` 段)能模糊匹配到某工单 → warn「目录 slug 与当前 workItemSlug 不一致(疑似底盘 slugify 升级),建议 `git mv <旧> <新>`」(吸收 O7 新形态,给迁移留缓冲,**不**误报「Done 缺 spec」)。
  - 完全匹配不到任何工单 → warn「孤儿 work 目录」。

**新增 `state/receipt-grandfather.json`**(模板,`{ "ids": [] }`);`--upgrade` 时由 init.mjs 回填当前 Done 工单 ID(见 1.5)。

### 1.3 新增 regression-diff.mjs(O17)

**新增 `bootstrap/workflow/scripts/regression-diff.mjs`** + package.json alias `regression:diff`:
- 基线文件 `state/flaky-baseline.json`:`{ "suites": [{ "name": "<test 文件名或 suite 名>", "reason": "<为何预存红>", "recorded_at": "<ISO8601>" }] }`(模板初始 `{ "suites": [] }`)。
- `npm run regression:diff -- --failures "a.test.js,b.test.js,..."`:把传入失败套件分两组输出 JSON:
  - `new`:不在基线 → 需举证(可能是真回归)
  - `known`:在基线 → 自动归类预存 flake
  - 退出码:`new` 非空 → `exit 1`(主线据此判定有新增失败);`new` 空 → `exit 0`(只剩预存 flake,收敛通过)。
- `npm run regression:diff -- --add "<name>" --reason "<text>"`:登记新 flake 进基线(主线在隔离重跑确认该套件单跑绿后方可登记;脚本本身不验,纪律靠 SKILL.md)。
- **MVP 边界**:脚本不解析 jest/vitest 原始输出(框架多、格式杂);主线从测试输出提取失败套件名以 `--failures` 传入。后续可加 `--parse <jest-json-path>` 自动提取,本轮 YAGNI。

### 1.4 status 脚本 --json 纯净化 + 窄查询(O23 部分)

**改 `e2e-group:status` / `milestone:status`(及 init.mjs 注册的同类):**
- `--json` 模式:**JSON 串只写 stdout**;所有人类可读 banner / 进度 / 提示写 stderr。(npm run 自身 banner 无法消除——故 SKILL.md 同时给「直调 node 脚本」的解析式。)
- 增 `--field <key>`:输出该字段裸值单行(如 `--field next_action` → `run_group_e2e`),主线一行 shell 即可消费,免即兴 grep/python。

### 1.5 init.mjs --upgrade + VERSION(分发)

**新增 `bootstrap/workflow/VERSION`**:纯文本语义版本,本轮 `5.4.0`。`--bootstrap` 与 `--upgrade` 都复制进 `<devRoot>/workflow/VERSION`。

**改 `bootstrap/workflow/scripts/init.mjs` 增 `--upgrade --target <devRoot>` 模式:**
- 幂等覆盖 `<devRoot>/workflow/scripts/` + `<devRoot>/workflow/templates/`(源 = skill 自带 `bootstrap/workflow/`)。
- **对照清单补齐** `<devRoot>/package.json` 必需 alias(start / regression:diff / validate:state / promote / deps:graph / lint:redlines / verify:handoff / milestone:status / e2e-group:status / render:board / start 等全套);已存在的不动,缺的补。
- 复制新 `VERSION`。
- 首次 upgrade 时若 `state/receipt-grandfather.json` 不存在或 ids 为空 → 从 queue.md 枚举当前 Done 工单 ID 回填(D4 祖父豁免,见 1.2)。
- **绝不碰** `state/*`(receipt-grandfather.json 除首次回填外)、`workflow.config.mjs`、`work/`。
- 结束打印摘要:`[upgrade] 覆盖 <N> 脚本/模板 · 新增 <M> alias · VERSION <old>→<new>`,并自动 `cd <devRoot>/workflow && npm run validate:state` 验证后报告。

**SKILL.md 启动协议增「底盘契约自检」一步**(契约包 2.x):定位 devRoot 后,比对 skill 自带 `bootstrap/workflow/VERSION` 与 `<devRoot>/workflow/VERSION`,或关键 alias(start/regression:diff)缺失 → 停下,引导用户**主线亲跑** `node <SKILL_ROOT>/bootstrap/workflow/scripts/init.mjs --upgrade --target <devRoot>`(与 bootstrap 同款:不派 sub-agent、不手 Write 底盘)。这是不变量 9 的升级闭环。

## 2. 契约包(SKILL.md + references/)

> 依赖底盘包的脚本行为。改 SKILL.md 集中一次,references/ 同步。

### 2.1 receipt 落盘契约(O13 核心)

**SKILL.md「Receipt 契约」段重写要点:**
- receipt 的**权威载体是落盘文件** `<devRoot>/work/<slug>/<receiptsDir>/<stage>.json`,**由该 stage 的 sub-agent 自己 `Write`**;末条消息中的 receipt 仅作冗余副本。
- 主线协议:派工返回后**第一动作** `test -f <receiptPath>`。不存在 = **交付失败**(走不变量 10 自愈:fresh 重派 1 次 → 内联接手),**不再依赖解析末条消息判交付成功**。
- 需从消息读 receipt 做冗余校验时,取「**最后一个合法 JSON 块**」而非严格末条(与 drafter 在 JSON 后追加散文的现实对齐,减少假交付失败)。
- 派工 prompt 用 `{{receiptPath}}` 占位把落盘绝对路径钉死,避免 sub-agent 自己猜路径。

**修订不变量 8**:`pipeline-status.json` 仍主线单写;**stage receipt 文件改由对应 sub-agent 写**,路径主线钉死。二者不冲突(不同文件、不同写者)。

**references/receipts-schema.md** 同步:envelope 说明落盘者 = sub-agent;新增「主线 `test -f` 校验」段。

### 2.2 arch-security-reviewer 收编(根治 O1)

- arch-security-reviewer **不再自产 receipt**。返回结构化 findings 段:`{ red_line_hits: [], security_findings: [{severity, file, desc}], verdict_suggestion: "READY_TO_HANDOFF|NEEDS_FIX", scope_check: "..." }`(作为消息内容,非 receipt 文件)。
- **4-arch receipt 由主线确定性拼装并 Write**:主线 Stage 4 = 亲跑 `lint:redlines`(含 redlineCommands)+ 读 sub-agent findings → 组装 `4-arch.json` 落盘。
- SKILL.md 质检表 Stage 4 行 + 「调用 sub-agent 的 prompt 模板位置」表同步该语义。
- 理由写入设计:6/4 的 f12cee7 中主线 6/6 都在手补 receipt——本改动把既成事实正式收编,消灭注定失败的 receipt 往返与「主线手造 receipt」审计失真。

### 2.3 E2E 两段式(O12,吸收 O20)

**SKILL.md 可选 E2E 线段(里程碑级 + 组级)重写为两段:**

**段一 · 展开 journeys 清单**:边界到达(`milestone:status`/`e2e-group:status` 报到达)后,主线**内联**(默认;旅程多/复杂可一次轻量派工)把 acceptance.md 该里程碑段 / 组成员 spec 的验收标准展开成 `<reportsDir>/journeys-<scopeId>.json`:
```json
{ "scope_id": "EG-...", "journeys": [
  { "journey_id": "J1", "desc": "...", "steps": ["..."], "status": "pending" }
] }
```

**段二 · 逐旅程短 verifier**:主线对每条 `status:pending` 旅程派一个独立 `e2e-verifier(mode=journey, journey_id=Jx)`:
- **截图落盘即弃**:截图写 `evidence/<scopeId>/<journey_id>/`,thread 内只留路径,**禁止把 base64 留在上下文**。
- 单旅程 receipt 落 `evidence/<scopeId>/<journey_id>.json`(verdict / 证据路径 / regression 结果)。
- 主线汇总各旅程 receipt 拼组级 receipt `<reportsDir>/e2e-<scopeId>.json`。

**断点续跑**:段二派工前扫 `evidence/<scopeId>/`,已 PASS 旅程跳过——中断后续跑不重做。

**第三类失败「policy/transport 硬杀」**:verifier 被 AUP/连接杀死无单旅程 receipt → 主线据该旅程已落 evidence **内联补**一个 `reason_category=env-blocked` 或 `escalated_to_human=true` 的旅程 receipt,让组有尾、不卡 Open(区别于不变量 10 的交付失败:这是不可重试硬墙,直接降级标注)。

**组级 receipt 可读性(吸收 O20)**:组级 receipt 顶层增:
- `acceptance_legible_status`:枚举 `ACCEPTED` / `ACCEPTED_WITH_ENV_BLOCKED` / `FAILED`
- `env_blocked_reason`:string|null
让不懂 b2r 协议的下游(人/CI)一眼判非绕过。`references/receipts-schema.md` 同步 e2e receipt schema。

### 2.4 infra 错误指纹协议(O18)

**SKILL.md 失败处理段增「infra 错误指纹」小节:**
- 指纹清单:`403 Request not allowed` / `Please run /login` / `socket connection ... closed` / `use SendMessage with to:` / `overloaded` / `529` / `Usage Policy` / 裸 `API Error`。
- 硬规则:sub-agent 返回命中任一指纹 = **交付失败**(归入不变量 10),主线**当回合**处理:先探盘上半成品(spec/plan/commit 是否已部分落盘)→ fresh 重派或内联接手。**禁止把错误原文回显给用户等「继续」**。
- 边界保留:进程级真 hang 仍测不了,依赖 harness 超时回收(不变量 10 既有诚实边界不变)。

### 2.5 主线亲核 gate(O16)

**SKILL.md 质检表 Stage 2 / Stage 3 增主线 gate 行:**
- **Stage 2 收 spec/plan receipt 后**:主线亲跑占位 grep `grep -nE 'TBD|待定|待起草|占位|待 *fresh' <spec/plan 文件>`,命中即判 sub-agent 谎报 sections_filled、打回(不信 receipt 的 `tbd_grep` 自报字段)。
- **Stage 3 收 impl receipt**:receipt 必含 `failing_test_output`(红阶段测试输出关键行,或 artifact 路径);主线核红 gate 凭此证据,**不凭** `failing_test_first:"pass"` 布尔。缺字段 = receipt 不合规,打回。

`references/quality-gates.md` 同步这两条主线 gate。

### 2.6 UI / E2E 线显式播报(O15)

**SKILL.md Stage 1 段:**
- roadmap-planner proposal 带回 `ui_intent_detected`(模板包改动,见 3)。主线见 `ui_intent_detected:true` **且** config 无 `ui` 块 → **必须 AskUserQuestion**:「检测到 N 条工单涉及前端,但 UI 设计线未配置 —— (a) 现在加 ui 块走完整前端 (b) 确认只做后端、前端 defer」。
- Stage 1 backlog 落地汇报**固定播报一行**:`UI 线:开/关 · E2E 线:开/关(unit=...)`,杜绝静默少交付。
