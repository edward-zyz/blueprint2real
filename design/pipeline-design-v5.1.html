Gate 失败 · 两层处理机制

Auto-Retry × 1 → Manager Override（取消独立的 escalate 阶段）

Gate fail 不直接 Halt 主线。v5 简化为两层： 最多 1 次 Auto-Retry，仍失败 或 sub-agent 自报"我搞不动了"，立即进入 Manager Override（无中间的"escalate 阶段"独立存在；escalation-pack.md 仅作为 Manager Override 入口的卷宗自动生成）。

L1
AUTO RETRY · max=1
Gate fail 第 1 次：把 fail_items 打包成 feedback-receipt.json，回路给上一个 stage 派 fresh sub-agent，明示"上一轮哪些条不通过 + reviewer 期望什么"。Sub-agent 看 feedback + 自己上轮 receipt，针对性修正而非重写。
注：sub-agent 在被派工时如自报"无法解题"（基于 prompt 中的"自报阻塞"约定），跳过 retry 直接进 L2。
默认 max_retry = 1
workflow.config 可调（暂保留 hook）
不打扰主线 thread
L2
MANAGER OVERRIDE
触发条件：retry 1 次仍 fail 或 sub-agent 自报阻塞 或 Gate 8 (handoff verify) 失败。
动作：主线自动生成 escalation-pack.md 卷宗 → 主线起草决策建议 → 用 AskUserQuestion 让用户拍板 → 写 manager-decision.json → 按决策强制推进，后续 receipt 带 manager_override 标记；同时追加 state/retro.md。
人决策 → 主线落地
5 种推进方式见 [08]
最终"卡不住"保障 + 体系改进证据

★ 设计意图：v4 原本设计的"L2 Escalate 阶段"在 max_retry=1 + 允许 sub-agent 自报阻塞 之后已经退化为单纯的"卷宗自动整理"——不再是独立处理阶段，所以合并到 L2 的准备步里。这避免了"系统自己反复尝试且不告诉人"的反模式。

Manager 决策选项 · 5 种强制推进方式

人介入后，所有后续 receipt 必须带 manager_override 标记
决策 语义 受影响
accept-override 接受现状强制 pass。当前 receipt 的 verdict 强行翻 READY，进入下一 stage。
适合：reviewer 给的是过度严格的"理论问题"但实际可接受		 下一 stage 接力执行；handoff 时 customer-visible.md 写明 "manager-override accepted on Gate5"
downgrade L3 降 L2 / L2 降 L1。跳过当前卡住的严格 stage（如 arch reviewer），用轻量版替代。
适合：发现工单实际复杂度低于初判，严格 stage 是过度设计		 跳过当前/后续严格 stage；customer-visible 记录降档
shrink-scope 人编辑 spec §3 "本轮不做" 把卡住的边界条款加进去，重新从 spec-drafter 走一次。
适合：发现工单边界没收住，把卡住的部分剥离		 把卡住的部分剥离成新工单（自动入 Planned queue）；本工单 spec §3 + §11 同步更新
split-slice 把工单拆成 sub-slice，先做能过 gate 的部分。
适合：工单 spec §4 范围过大，部分能做、部分卡住		 plan §3 加 Sub-slice 列表；implementor 按 slice 重新启动
drop 标记工单 Superseded（编号不还回池）。
适合：发现工单本身错了，前置假设不成立		 queue 行 Superseded；active.md 翻 Idle；customer-visible 写"暂停 + 原因"

主线 thread 落地流程（v5 · 主线起草 + 用户确认）

  1. Gate fail attempt 2（或 sub-agent 自报阻塞 / Gate 8 fail）→ 主线接到 sub-agent return + 失败上下文
  2. 主线自动整理 escalation-pack.md 卷宗（含历次 receipt diff / sub-agent 自报阻塞点 / 历次 feedback）
  3. 主线自动起草决策草案（基于卷宗 + 5 选 1 的语义匹配）：「我建议：<option> · 理由：... · 影响范围：...」
  4. 主线调 AskUserQuestion 呈现草案 + 4 个可选行动给用户：
    • Accept 推荐
    • 改选其他 option（4 个）
    • 微调 action_params（用户用自然语言反馈，主线再修订草案）
    • Drop / Abort
  5. 用户拍板后主线写入 dev/work/<id>/receipts/manager-decision-<timestamp>.json
  6. 主线追加 dev/state/retro.md 一段（失败链 + 决策 + 1 行经验）
  7. 主线按 action 字段调度：
    • accept-override → 直接派下一 stage sub-agent，prompt 含 manager_override 标记
    • downgrade → 改 receipts/0-triage.json 中 level，重新走 Gate（按新 level 分支）
    • shrink-scope → 派 spec-drafter retry，prompt 含"必须加 section3 不做"指令
    • split-slice → 派 plan-drafter retry，prompt 含"声明 sub-slice"指令
    • drop → 直接走 handoff 流程，queue 翻 Superseded
  8. 所有后续 receipt 必须带 manager_override: { gate, decision_path } 字段；handoff 时由 verify-handoff 检查并要求 customer-visible.md 明示

★ 为什么不用手编 JSON / CLI：进入 Manager Override 时主线上下文已经齐全（卷宗、卡点、所有 receipt 都在），起草成本几乎为零；用户负担降到"yes / 改选项 / 自然语言反馈"。手编 JSON 容易出 schema 错；CLI 多轮问答比一次成稿见全貌差。
