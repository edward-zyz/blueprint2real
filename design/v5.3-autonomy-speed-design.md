## 1. 背景与目标

两天里 b2r 跑通 IS-017~030（doc-hub M1/M2），**全部 attempt=1、零 Manager Override**。pipeline 已经基本能跑——这是在一个**成熟**系统上做**边际硬化**，不是补窟窿。诉求收敛成一句话：

> **无人值守 + 不降交付质量 + 顺带快一点。**

### 唯一不可妥协的原则

**质检 gate 一条都不松。松的不是"判定标准"，是"卡住之后要不要停下来等人"。** 质量靠 gate 保住（TDD 红→绿、范围⊆spec§4、lint:redlines、verify:handoff 七项——判定原文不动）；自主靠**对"没拿到可用产出"自动恢复**拿到。任何为"让它别停"而放宽 gate 判定的改法，不接受。

### 诚实的能力边界（reviewer P0 的结论）

主线派 `Agent` 是**同步阻塞**等待返回的。sub-agent 若**真 hang**（进程级无限静默），主线自己也一起 hang——它没有旁路 watchdog、数不到"N 分钟"、无从触发接管。**所以本设计不治真 hang**，那只能靠 harness 侧的 Agent 超时把控制权交回主线。本设计**只治"主线拿到了返回、但返回不可用"**这一可观测子集（529/overloaded 错误串、空返回、截断、末条是散文非 JSON）。这条边界必须写死，否则"无人值守"是空头支票。

## 3. 核心说理：区分"两类失败"

当前 SKILL「失败处理」只建模了**一类**失败。把两类显式分开，是 Item 1 的支点。

| | 质量失败（gate fail） | 交付失败（returned-but-unusable）← 新增 |
|---|---|---|
| 定义 | receipt 合法，但 verdict=NEEDS_FIX / 脚本非 0 / 范围越界 | Agent **返回了**，但内容是 529/overloaded 错误串、空、截断、或末条散文非 JSON |
| 现状 | ✅ 已建模：retry-once → Manager Override | ❌ 未建模：主线要么硬塞错误产物，要么临场手工补 |
| 处理 | 不变（质量分歧该让人拍板） | **receipt 兜底自愈**：fresh 重派 1 次 → 仍不行主线内联接手 → 再不行才 Manager Override |
| **不在范围** | — | **进程级真 hang**：主线同步阻塞无从检测，依赖 harness 超时回收，本设计不保证 |

关键区别：**质量失败该升级到人**；**交付失败（可观测子集）不该惊动人**，自动恢复即可。

## 4. 最终三项（+ 一项轻量）

> 全是 agent 模板文字 + SKILL.md 路由段改动，**零新脚本、零新 config 字段**。

### Item 1 · receipt 合法性兜底 + 自愈 ←【无人值守核心】

**问题**：Agent 返回不可用产物（错误/空/截断/散文）时，主线无明文协议，实测靠临场手工补（IS-024 slice2、IS-026 Stage 4 主线复跑）。

**落点**：
- `agents/*.md` 全 6 份「返回格式」段**统一**硬约束（把 `arch-security-reviewer.md`:93 那段复制到缺的 4 份）：
  ```
  **硬约束**：最后一条消息必须是本 stage 的 receipt JSON，散文报告放 JSON 之前。
  只给散文不给 JSON / 返回报错或空 = 交付失败，主线按 receipt 兜底协议自动处理，不计入 gate attempt。
  ```
- `SKILL.md`「失败处理」段新增「交付失败兜底」小节 + 新增不变量 **10**（§5）。
- `references/quality-gates.md` + `pipeline-flow.md` 补「交付失败」虚线分支。

**协议**（主线每次 Agent 返回后执行）：
```
解析末条消息：能解析成本 stage 的 receipt envelope（stage_id/level/attempt 齐）→ 可用，进既有 gate 判定。
否则 = 交付失败，不惊动用户：
  1. fresh 重派同一 stage 一次（prompt 末附"上次未返回合法 receipt"）。这步不计入 gate attempt。
  2. 仍不可用 → 主线内联接手该 stage，落 receipt，标 dispatch_recovery（不复用 manager_override 字段）。
     · 内联代跑 implementor 时，照样受不变量 3（TDD 红→绿）/ 4（commit 物理分离）约束，红 gate 证据照常落 3-impl.json——不因主线接手而豁免。
     · 内联接手有次数上限（见 §7 验收）：超限本身升级 Manager；接手产物落盘 receipt 后不长留主线对话，避免长链上下文污染。
  3. 主线内联也做不动（真·能力边界）→ 才进 Manager Override（人）。
```

> **为什么用 `dispatch_recovery` 而非 `manager_override`**（架构 reviewer P1）：`manager_override` 字段语义＝"经过 AskUserQuestion + manager-decision.json + retro 的人介入流程"。内联接手**没有人**，复用该字段会让 retro surface / BOARD 把自愈误统计成真 Override，污染审计。

### Item 2 · 单切片 L2/L3 合并 2a+2b ←【最干净提速】

**改动**：预判 sub-slice=1 且未走"plan 已存在捷径"时，一次合并 drafter 调用同 context 产出 `spec.md`+`plan.md`。`spec-plan-reviewer` **照常跑、不可省**（L2/L3 铁律）。

**落点**：`SKILL.md` Stage 2 + `pipeline-flow.md`「4 档路径」加单切片合并旁路；现有 `spec-drafter.md`/`plan-drafter.md` 顶部加"可合并触发条件"。

**收益/风险**：单切片省一次 dispatch 往返；独立 reviewer 仍兜质量，低风险。

### Item 3 · regression 收敛到 handoff 前

**改动**：implementor 切片内**只跑 targeted test**（必绿）；全量 `regressionCommands` **收敛到最后一个 slice 后、Stage 4/handoff 前跑一次**。

**落点**：`SKILL.md` Stage 3 质检表 + `references/quality-gates.md`:23-24（Stage 3 regression 行）+ `implementor.md`:190（主线收尾改 targeted）——**三处必须同步改**，否则 regression gate 归属两处矛盾（架构 reviewer P1）。

**风险 + 必带缓解**（红队 P1）：多切片工单 slice2 可能破 slice1 全仓测试，收敛点才暴露、而 implementor context 已散。**所以收敛 regression 红时必须能按 slice 二分定位**——保留每 slice 的 targeted + 一个增量集成 checkpoint，而非裸跑全量面对红海。缓解到位才允许上 Item 3。

**收益**：多切片工单 regression 墙钟 N× → 1×（单切片无变化）。

### Item 4（轻量）· retro 模板加 `template_patch` 字段 ←【人值守复盘增强，非无人值守能力】

**诚实定位**（红队 P1）：A2 初稿想"闭环经验自动回灌"，但软校验只 warn、回灌仍靠人——无人值守下没人看 warn，与它要修的"经验没人回灌"**同构**。所以**砍掉 validate-state 软校验脚本**，只留最便宜的一半：

- `bootstrap/workflow/templates/state/retro.md.tmpl` 每条 override 加一行 `- template_patch: <agents/xxx.md, ...> | none`。
- `SKILL.md`「Retro 复盘机制」：Surface 触发（每里程碑 / ≥3 条）时，主线把 `template_patch != none` 的条目汇总成"待回灌清单"给用户。

**标注**：本项**只服务有人值守的定期复盘**，让回灌债可追踪；不进"无人值守最小闭包"。实际改 `agents/*.md` 仍是人监督下的 `/skill-review`。

|---|---|---|---|
