## 逐项详情

### Item 1 · receipt 兜底自愈

- **原因**：主线派 `Agent` 后若返回 529/空/截断/散文（非合法 receipt），当前 SKILL 无协议——实测靠主线临场手工补（IS-024 slice2、IS-026 Stage 4 复跑）。无人值守模式下这是静默卡死点。根因是"失败处理"只建模了"receipt 合法但没过 gate"一类，漏了"根本没拿到可用 receipt"一类。
- **方案**：SKILL.md 新增**不变量 10**，把失败分两类——
  - 交付失败（returned-but-unusable）→ **机器自愈**：fresh 重派 1 次 → 仍不可用主线内联接手（标 `dispatch_recovery`，不复用 `manager_override`，内联代跑 implementor 仍守 TDD/commit 不变量）→ 才 Manager Override。不计 gate attempt。
  - 质量失败（receipt 合法没过 gate）→ 维持原 retry-once → Manager。
  - 落点：`SKILL.md`（不变量 10 + 交付失败兜底小节）、`references/quality-gates.md`（dispatch 行）、`references/pipeline-flow.md`（三类失败一图）、**8 份 `agents/*.md`** 返回段统一硬约束"末条必须 receipt JSON"。
- **预期结果**：无人值守跑一条 milestone 工单链，**主线手工接管次数趋零**（可证伪版：每次内联接手必留 `dispatch_recovery` receipt + 不超次数上限）。
- **诚实边界**：**进程级真 hang 测不了**——主线同步阻塞等 Agent，无 watchdog，依赖 harness 超时回收。本项只治"返回了但不可用"。
- **成本**：零新 config；主线内联接手会临时多背上下文（用"次数上限 + 产物落盘后不长留对话"约束）。

### Item 2 · 合并 2a+2b（单切片）

- **原因**：L2/L3 恒定拆 2a-spec / 2b-plan 两次独立 dispatch。实测单切片工单这俩只隔 ~4min（IS-026/027/030 的 2a→2b 间隔），各是一次 drafter 往返，但产物天然同 context。
- **方案**：triage/spec 预判 **sub-slice=1** 且未走"plan 已存在捷径"时，派 `spec-drafter` 标 `merge_2b:true`，让它产出 spec 后**同 context 续写 plan**，一次返回两份 receipt。`spec-plan-reviewer` **照常跑、不可省**（L2/L3 铁律）。落点：`SKILL.md` Stage 2 旁路。
- **预期结果**：单切片工单 Stage 2 的 sub-agent dispatch **2→1**（约省一次往返）。
- **实测验证**：IS-027、IS-030 均单切片仍拆 2a→2b——新版各省一次 dispatch。
- **风险**：低。独立 reviewer 仍兜质量；多切片/拿不准则不合并，老路分派。

### Item 3 · regression 收敛到 handoff 前

- **原因**：Stage 3 每个 slice 都跑 `config.regressionCommands`（含全仓 `npm test`）。多切片 = N 次全量跑，绝大部分无关。
- **方案**：implementor 切片内**只跑 targeted**（必绿）；全量 `regressionCommands` **收敛到末切片后、Stage 4 前由主线跑一次**。收敛红了**按切片二分定位**（每切片留 targeted + 增量集成 checkpoint）。落点：`SKILL.md` Stage 3 质检表、`quality-gates.md`、`implementor.md`。
- **预期结果**：多切片工单全量 regression **N×→1×**（单切片无变化）。
- **实测验证**：IS-029（2 切片）slice1+slice2 的 `regression_results` 都含 `npm test`——跑了 2×，新版收敛 1×。
- **风险**：集成 break 推迟到收敛点暴露——用"收敛仍是硬 gate + 按切片二分"兜底，质量不丢。

### Item 4 · retro `template_patch` 字段

- **原因**：retro 记下的"1 行经验"若指向某 `agents/*.md`，**无机制保证回灌**——同一 handoff 自指 hash 坑跨 14 天复发多次。无人值守时更致命：没人盯着，复发坑安静产坏 handoff。
- **方案**：retro 模板每条加 `- template_patch: <agents/xxx.md … | none>`；Surface（每里程碑/≥3 条）时主线把 `!= none` 的条目汇总成**待回灌清单**呈现。落点：`bootstrap/.../retro.md.tmpl` + `SKILL.md` Retro 机制。
- **预期结果**：指向模板的经验在复盘时 **100% 进待回灌清单**，回灌债可追踪。
- **诚实边界**：**只服务有人值守的定期复盘**——实际改模板仍走 `/skill-review`；无人值守模式没人看清单，故不属"无人值守能力"。砍掉了初稿的 validate-state 软校验脚本（warn 无人看，与原病同构）。

### ★ direct-fix 老坑补修（顺手清的历史债）

- **原因**：`8a641bc2` 修了 handoff-committer 的自指 hash + amend handoff 坑，但**漏改 `direct-fix.md`**——它 Step 5 仍"回填 active.md Last commit + amend handoff commit"，下个 L0 工单会产生 dangling commit。这正是 Item 4 要治的"回灌不彻底"活体样本。
- **方案**：比照 handoff-committer v2——Step 4 active.md `Last commit` **一次写定 impl hash**、Step 5 handoff **一次成型不 amend**、禁项改"不 amend 任何 commit"。
- **预期结果**：L0 收尾后 active.md/BOARD 指向**有效 impl hash**，不再有悬空 commit；`8a641bc2` 覆盖面补齐（handoff-committer + direct-fix 两条收尾路径口径一致）。

### Item 5 · 删除/退役类工单 spec §4 计数对账

- **原因**：IS-029（showdoc **退役**）spec 叙述写"前端 8 / 共 14 / git rm 14"，磁盘实为 7 / 13，reviewer attempt-1 打回 → **重跑整轮 spec+plan+review**。根因＝spec-drafter 的 §4 自检只查"文件存不存在"，对删除场景失效（失效模式是"计数准不准、叙述与清单一不一致"）。
- **方案**：`agents/spec-drafter.md` b2r 约束 #4 增补 + 自检第 5 条——删除/退役/cleanup 类工单 §4 删除清单**从 `git ls-files <目录>` 实际枚举生成**（不靠 roadmap 估数）+ **叙述数字与 itemized 列表逐一对账**，不一致先改对再返回。
- **预期结果**：删除类工单避免"计数漂移触发整轮重跑"（IS-029 实测代价＝白跑一轮 spec+plan+review）。
- **诚实边界**：**n=1 的廉价护栏**——删除/退役/cleanup 是会反复出现的工单类别，但目前只一条实例；只加了两句 prompt 文字、零新机制，符合"别为 n=1 过度工程"。

## 被砍 / 否决（一句话，详见 design §6/§8）

- **砍**：A1 的超时/watchdog 措辞（主线同步阻塞测不了真 hang）、B1 Stage4 inline 降级（谓词判错代价 > 收益）、C1 多镜 fan-out（零漏检实例，YAGNI）、A2 的 validate-state 软校验脚本（warn 无人看）。
- **否决**：工单并行（撞单 active/单写者/commit 分离不变量，且自主诉求不需要）、state DB 化（SQLite 库级串行写、二进制破 git 审计；分片应用目录不用 DB）。
