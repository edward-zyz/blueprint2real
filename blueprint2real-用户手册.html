这是什么

一句话：会自我把关的"AI 开发团队调度器"

blueprint2real（简称 b2r）是一个 agent skill——装进你的 AI 编程助手（如 Claude Code）后，它不亲自闷头写代码，而是当一个项目经理：把你的路线图接进来，按固定流程调度一批各司其职的 sub-agent，并在每个环节用脚本强制检查，最后交给你一条状态清晰、提交干净、可被事后审计的"Done 工单"。

📐 名字的含义：blueprint（蓝图：你的 roadmap / 设计文档） → real（现实：真正合并、可演示、可审计的代码）。它专门解决"想法很多、落地很慢、过程说不清"这件事。

🔑

它不是一个跑代码的服务，而是一套约定 + 脚本 + prompt。 全部基于本地的 Markdown、JSON、Shell 和 Node.js 文件运行，不依赖任何托管服务或专有 API。你可以完全看懂、完全掌控、随时回滚。

使用场景

你是不是也遇到过这些

当你用 AI 助手做真实项目、而不只是写个小脚本时，"对话式开发"的问题就会冒出来。

😖 没有 b2r 时

  • roadmap 写了一堆，spec 起草拖好几天
  • 开发跟 spec 渐渐漂移，做出来跟说好的不一样
  • QA 永远在追赶，没人说得清"这周到底交付了什么"
  • AI 改动散落在一次次临时对话里，过程无法复盘
  • 一次 commit 把代码、状态、看板混在一起，想回滚都不知道滚哪段
  • AI"假装"做完了多件事，其实状态自相矛盾

✅ 用了 b2r 后

  • roadmap 一进来就被拆成带依赖、带复杂度标签的工单序列
  • spec / plan / 实现严格对齐，越界会被脚本拦下
  • 每条工单有完整证据链（receipt），随时可审计
  • 看板（BOARD.html）由脚本生成，永远反映真实状态
  • 实现提交与状态提交物理分离，任意一段可独立回滚
  • "同时只能有一条工单在做"由脚本强制，杜绝幻觉

读懂原理 · 1

核心思想：把"聊着写代码"变成"流水线工厂"

理解 b2r 最好的比喻，是一座汽车装配工厂。这个比喻能一次性讲清它所有的设计。

🏭

工位 = Stage（阶段）

整车不是一个人从头造到尾，而是分成若干工位：起草设计 → 实现 → 检查 → 交付。每个工位职责单一。

👷

专职工人 = sub-agent

每个工位站着一个专门的 AI 工人，只看自己需要的图纸（spec/plan），干完活就交一张工单回执，不和别人闲聊。

🔍

质检员 = 脚本门禁（Gate）

每个工位下线前必须过质检。质检不是工人自己说"我觉得行"，而是跑一段脚本，不通过就不放行。

🧾

质检单 = receipt

每个工位都留下一张结构化回执（JSON）。所有回执串起来，就是这条工单完整、可审计的证据链。

💡

关键洞察：单个 AI 容易"幻觉自己做完了"。b2r 的破解之道是——不相信任何 agent 的口头汇报，只相信脚本的退出码和文件里的证据。所有"是否通过"的判断，都由确定性的脚本做，而不是由 LLM 自评。这就是它能把"差不多得了"变成"真正 Done"的根本原因。

读懂原理 · 2

三大支柱：事实源、底盘、调度者

b2r 的整个系统建立在三个分工明确的层上。看懂这三层，就看懂了它的架构。

📄
支柱一 · 唯一权威

事实源 state/*.md

state/ 里的 Markdown 文件是唯一权威：当前在做什么、队列里有哪些工单、路线图、客户可见变化……看板 BOARD.html 只是渲染产物，从不手改。冲突时一切以 state/* 为准。

⚙️
支柱二 · 物理实现

底盘 workflow/scripts

一组 Node.js 脚本：校验状态、生成看板、推进工单、核验交付、检查红线……它们把"规则"变成可执行的代码。底盘只能由初始化命令一次性生成，不允许 agent 临时重造，避免版本漂移。

🧠
支柱三 · 编排大脑

调度者 the skill

就是 b2r skill 本身。它读取配置，按固定执行链路调度 sub-agent，在每步之间亲自跑脚本兜底，处理失败与升级。它是把"底盘"真正驱动起来的那个大脑。

⚠️

为什么要分这三层？ 因为"规则"如果只写在 prompt 里，LLM 会慢慢忘、慢慢飘。把规则下沉成脚本（底盘），把状态固化成文件（事实源），LLM 只负责调度与创作（调度者）——这样即使换一个 AI 模型来跑，约束依然成立。这就是它"Agent 中立"的底气。

读懂原理 · 3

Agent 角色团：11 个各司其职的工人

每个角色都是一份 agents/<role>.md prompt 模板。调度者派工时把工单的具体信息填进占位符，启动一个全新上下文的 sub-agent。角色之间互不读对方的内部推理——比如评审员看不到起草者的思考过程，这样评审才是真正独立的"第二双眼睛"。

🗺️

把 roadmap 拆成带依赖的 Planned 工单，并给每条打复杂度标签（Stage 0/1）。

agents/spec-drafter.md

什么时候派 sub-agent，什么时候调度者亲自做？ 需要全新上下文的创作（起草 spec、TDD 实现）、需要独立判断的评审 → 派 sub-agent。而跑脚本、读状态、和你对话、起草升级决策 → 调度者自己做。sub-agent 永远不会再启动 sub-agent，保持单层调度，责任链清晰。

读懂原理 · 4

六阶段流水线：一条工单怎么走完

这是 b2r 的固定执行链路。每个阶段都有专职执行者和必须通过的脚本门禁（绿色标签）。

0

Triage 复杂度分诊 roadmap-planner

给每条工单打 L0–L3 标签，决定它走"完整 / 中等 / 简化 / 单工位"哪条路。判据写入 0-triage.json，只能升档不能降档。

1

Roadmap → Backlog roadmap-planner validate:state

把 roadmap 拆成 Planned 工单序列（含依赖）。真实工单号由调度者脚本化分配，不让 AI 手算。

2

Promote 推进为 Ready spec-drafter + plan-drafter + reviewer validate:state · deps:graph

把一条 Planned 翻成 Ready：生成 spec、plan、context-pack，并经独立评审。前置依赖必须全部 Done 才能 promote。

3

Implement 实现 implementor 红 gate · targeted · regression

工单翻 In Progress，TDD 红 → 绿：先有跑红的测试做证据，再写最小实现，最后定向测试 + 回归全过。

4

Review 架构与安全评审 arch-security-reviewer lint:redlines

L3 走完整评审，L2 走轻量内嵌，L1 跳过。红线 lint 命中即停，不带病往下走。

5

Commit & Handoff 交付 handoff-committer verify:handoff

实现提交（不含状态文件）+ 状态翻档 + 交付提交（只含状态文件）。两段物理分离，可独立回滚。最后核验交付完整性。

🔁

Stage 5 之后，工单回到 Idle，调度者亲跑 milestone:status 判断是否到了里程碑边界——没到就回 Stage 2 拿下一条；到了且配置了 E2E，就进入循环之外的里程碑端到端验收。整个过程不靠"感觉"，全靠脚本报到边界。

读懂原理 · 5

四档复杂度：小事走快道，大事走全程

不是每条工单都要走完整六阶段。Stage 0 给工单分诊到 L0–L3，越简单的工单跳过越多环节，把精力留给真正复杂的改动。

档位判据（满足任一）走的路径跳过
L0 极简typo / 注释 / 文档措辞 / 单行格式化direct-fix 单 agent：改 + 翻档 + 提交spec / plan / review / arch 全跳过
L1 简单单文件 + 无新接口 + 无 schema + ≤30 行spec/plan 合并 + 自评 → 实现 → 交付独立 reviewer、arch-reviewer
L2 标准跨 2–3 文件 + 有新函数 + 无跨模块边界变化spec → plan → 独立 review → 实现 → 轻量评审 → 交付reviewer 走轻量清单，不调外部 skill
L3 复杂跨模块 / 新 schema / 新依赖 / 安全敏感 / 含 migration完整七工位全程—
⬆️

只升不降：如果实现时发现改动其实比初判复杂（比如动了 schema），门禁会强制升档，重新走更严格的路径；但绝不允许把复杂工单偷偷降级成简单路径蒙混过关。

读懂原理 · 6

质检门禁与失败处理：出错了怎么办

b2r 把失败分成两类，区别对待——这是它既"自动化"又"不失控"的关键。

每个阶段的脚本门禁

调度者在每个 sub-agent 完成后，亲自跑脚本确认通过才进下一阶段。sub-agent 自己说"通过了"不算数。

阶段质检命令通过判据
Stage 1 Backlognpm run validate:state0 error
Stage 2 Promotevalidate:state + deps:graph0 error，依赖图无环无孤儿
Stage 3 红 gate核 3-impl.json有"测试先跑红"的证据
Stage 3 回归config 里的 regressionCommands全部退出码 0
Stage 4 Reviewnpm run lint:redlines0 命中
Stage 5 Handoffnpm run verify:handoff <id>全部检查通过

失败的三级处理

🩹

① 交付失败 → 自愈

sub-agent 返回了但产物不可用（报错串/空/截断）。这是基础设施问题，不惊动你：自动重派一次 → 仍不行调度者亲自接手。

🔁

② 质量失败 → 自动重试 ×1

产物合法但没过 gate。调度者把"哪条不通过 + 期望"打包，派一个全新 sub-agent 针对性修正，再过 gate。

🙋

③ Manager Override → 你拍板

重试仍失败，或 sub-agent 带证据自报阻塞。调度者渲染失败卷宗、起草建议，用选择题请你决策并留下决策记录。

🚫

绝不静默循环：重试失败必定升级到你面前，让你看到完整失败链——不会出现 AI "再试一次又一次"假装搞定的情况。Manager 的 5 个决策动作（接受 override / 降档 / 收缩范围 / 拆切片 / 放弃）各有明确回流点，全程留痕在 retro.md。

读懂原理 · 7

十条铁律：违反就当场停下

这些是从大量实战踩坑沉淀下来的硬约束。任何 sub-agent 在任何阶段命中其中一条，必须立即停下并报告，不允许"先记下、下轮再说"。它们是 b2r 不退化成"看心情写代码"的护栏。

#铁律它在防什么
1同时只有一条活跃工单（exactly one active）防 AI "假装在同时做好几件事"的幻觉
2事实源单一：state/* 权威，BOARD.html 只能脚本生成防状态与看板各说各话
3TDD 强制顺序：测试先跑红，才能写实现防"先写实现再补测试"的退化
4commit 物理分离 + 工作目录隔离让任一切片可独立回滚，防 cwd 泄漏到父仓
5前置 Done 才能 promote防把"想象的接口"当真接口去 spec
6架构红线触发即停不带病往下走
7自报阻塞必须带证据（grep/测试输出/条款）防空口"搞不动"式偷懒早退
8pipeline-status.json 由主线单写防 sub-agent 与主线并发写文件冲突
9skill 不直接写底盘脚本，只能 bootstrap 分发防 LLM 每次"贴心重造"导致版本漂移
10交付失败 ≠ 质量失败，前者自愈不惊动你区分基础设施抖动与真正的质量问题
动手使用 · 第一步

安装：把 skill 装进你的 agent

b2r 是一个 agent skill，安装就是把仓库克隆到你的 agent runtime 的 skills 目录。

克隆到 skills 目录

# 把 SKILLS_DIR 换成你的 agent runtime 实际的 skills 目录
SKILLS_DIR=/path/to/your/agent/skills
mkdir -p "$SKILLS_DIR"
git clone https://github.com/edward-zyz/blueprint2real.git "$SKILLS_DIR/blueprint2real"

安装底盘脚本依赖并自测

cd "$SKILLS_DIR/blueprint2real/bootstrap/workflow"
npm install
npm test     # 跑底盘脚本的单元测试，确认环境就绪

重启 agent session

开启一个新的 agent 会话，或在你的 runtime 支持时手动刷新 skills，让 b2r 被识别。

✅

装好后，当你说"开始 IS-001""promote 这条工单""从 roadmap 拆工单"等，或在含 b2r-process/ 的仓库里要求"按工单走一遍"时，agent 会自动触发这个 skill——你不必每次都念出名字。

动手使用 · 第二步

初始化：在你的项目里生成工作区

在你要管理的目标项目根目录下，运行一次 bootstrap，生成 b2r-process/ 工作区。这一步是底盘的唯一权威分发途径。

运行 bootstrap 命令

node /path/to/blueprint2real/bootstrap/workflow/scripts/init.mjs \
  --bootstrap \
  --prefix APP # 工单号前缀，2-4 个大写字母，如 IS/APP/WORK \
  --milestones M0,M1,M2 # 里程碑，逗号分隔；不用则留空 '' \
  --project myapp # 项目名（小写） \
  --target ./b2r-process

它会一次性生成 33 个文件：完整的 workflow/ 子树、workflow.config.mjs 配置、AGENT_RUNBOOK.md、初始 state/*.md 文件，以及带 npm 别名的 package.json。

安装工作区依赖

cd ./b2r-process/workflow
npm install

验证工作区

cd ./b2r-process
npm run validate:state   # 校验 state/* 结构，应当 0 error
npm run render:board     # 生成 BOARD.html 看板
⛔

不要手写底盘文件！（铁律 #9）如果 agent 发现 workflow.config.mjs 不存在，它的正确动作是停下来引导你跑上面的 bootstrap 命令——而不是"贴心地"自己创造脚本。底盘只能从 skill 自带的 bootstrap/workflow/ 一次性分发，保证大家用的是同一个权威版本。

动手使用 · 第三步

跑工单：用自然语言驱动流水线

初始化完成后，剩下的事几乎都用自然语言对 agent 说就行。下面是典型对话。

① 从 roadmap 拆工单（Stage 1）

# 你对 agent 说：
用 blueprint2real，把这份 roadmap 拆成 Done 工单。
# 或直接：
Use $blueprint2real to turn this roadmap into Done work items.

agent 会派 roadmap-planner 拆出带依赖、带复杂度标签的工单序列，回填真实工单号，并跑 validate:state 把关。

② 推进某条工单到 Ready（Stage 2）

# 你对 agent 说：
promote APP-001

生成 spec / plan / context-pack，经独立评审，确认前置依赖都 Done。

③ 开始实现 → 交付（Stage 3–5）

# 你对 agent 说：
开始 APP-001
# 完成后：
handoff APP-001

agent 走 TDD 实现、（按复杂度）评审、干净提交，最后跑 verify:handoff 核验交付。

④ 随时查看进度

cd ./b2r-process
npm run render:board    # 刷新看板，浏览器打开 BOARD.html

常见问题

FAQ

Q：我必须用 Claude Code 吗？

不必。b2r 是 Agent 中立的——它只依赖 Markdown、JSON、Shell 和 Node.js。只要你的 agent runtime 支持 skill 机制并能调用 sub-agent 与跑脚本，就能用。

Q：纯后端 / CLI 项目能用吗？UI 和 E2E 是必须的吗？

能用。UI 设计线和 E2E 验收线都是可选的——只有在 workflow.config.mjs 里配置了 ui / e2e 块才启用。缺省即关闭，纯库/CLI 项目不会因此报错。

Q：旧项目用的是 dev/ 目录，还能兼容吗？

可以。默认工作目录是 b2r-process/（v5.1 起），但 skill 会按顺序回退查找 legacy 的 dev/ 路径，两者都支持。

Q：BOARD.html 能手动编辑吗？

不能。它是 render:board 的渲染产物，手改会被下次渲染覆盖。要改状态，请改 state/*.md 这个事实源。

Q：为什么不让 AI 自己一次性把所有工单都 promote？

因为铁律 #5：前置依赖全部 Done 才能 promote。否则后面的 spec 就是基于"想象中的接口"写的，必然失真。一次只推进一条，是为了让每条 spec 都基于真实已交付的代码。

Q：出了问题，AI 会不会自己瞎搞？

不会无限瞎搞。质量失败最多自动重试一次，仍不过就升级到 Manager Override——渲染完整失败链、起草建议，用选择题请你拍板，并把决策记录留痕在 retro.md。绝不静默循环。

Q：怎么做整体验证？

# 仓库级一键验证
npm run validate
# 或分别跑
python3 .github/scripts/quick_validate.py .
cd bootstrap/workflow && npm test
bash evals/run-all-assertions.sh
