这是什么

一句话:会自我把关的"AI 开发团队调度器"

blueprint2real(简称 b2r)是一个 agent skill——装进你的 AI 编程助手(如 Claude Code)后,它不亲自闷头写代码,而是当一个项目经理:把你的路线图接进来,按固定流程调度一批各司其职的 sub-agent,并在每个环节用脚本强制检查,最后交给你一条状态清晰、提交干净、可被事后审计的"Done 工单"。

名字的含义:blueprint(蓝图:你的 roadmap / 设计文档)→ real(现实:真正合并、可演示、可审计的代码)。它专门解决"想法很多、落地很慢、过程说不清"这件事。

它不是一个跑代码的服务,而是一套约定 + 脚本 + prompt。全部基于本地的 Markdown、JSON、Shell 和 Node.js 文件运行,不依赖任何托管服务或专有 API。你可以完全看懂、完全掌控、随时回滚。

使用场景

你是不是也遇到过这些

同样是用 AI 做产品,你是"陪着 AI 写代码的人",还是"给 AI 派活的老板"?差别就在这里。

以前:跟 AI 结对编程

  • 你既是产品又是开发——AI 写一行、你盯一行,人一刻都离不开
  • 手里项目一多,你在好几个 AI 会话之间反复横跳、来回纠偏
  • 越用越累:不是 AI 在帮你干活,是你在疲于伺候 AI
  • 一天下来精疲力尽,感觉自己被 AI "榨干",而不是被解放
  • 过程散落在一次次临时对话里,交付了什么、能不能回滚,谁也说不清

现在:把蓝图交给 AI 团队

  • 你是老板:把想要的产品讲清楚,输出一份蓝图
  • 剩下的交给 AI 团队——拆活、实现、自检、交付,全程自走
  • 半小时把产品设计清楚,AI 跑一整晚
  • 早上回来,拿到一个可运行的产品,你只做验收
  • 每条工单都有完整证据链,做了什么、怎么回滚,一清二楚

本质是一次角色升级:从"陪 AI 写代码的开发者",变成"给 AI 研发团队派活的老板"。你负责想清楚要什么,b2r 负责把它真正做出来。

适合谁

用 AI 做真实项目的开发者

希望 AI 的产出可控、可追溯,而不是一团乱麻的对话历史。

需要交付证据链的团队

规划、实现、评审、交付都要留痕,方便审计与协作交接。

把 roadmap 系统化落地的项目

有明确里程碑、希望按工单一条条稳扎稳打推进的工程。

优势与特色

六个让它与众不同的设计

Roadmap → 工单流水线

把路线图自动拆成 Planned / Ready / In Progress / Done / Blocked / Superseded 等状态明确的工单,附带依赖关系。

分阶段 Agent 角色

规划、起草 spec、起草 plan、实现、架构/安全评审、直接修复、交付提交——每个角色都有专门的 prompt 模板。

默认开启的质量门禁

校验状态、依赖、架构红线、receipt、看板与交付边界。不靠 agent 自觉,靠脚本说话。

TDD 与提交纪律

强制"先写跑红的测试,再写最小实现",并把实现提交与状态/看板提交物理分离。

里程碑 E2E 验收(可选)

里程碑全部完成后,可按验收标准跑蓝图级端到端旅程,产出人类可读的验收报告。

任意 repo 可初始化

一条命令生成 b2r-process/ 工作区;纯 Markdown/JSON/Node 实现,Agent 中立、本地优先。

读懂原理 · 1

核心思想:把"聊着写代码"变成"流水线工厂"

理解 b2r 最好的比喻,是一座汽车装配工厂。这个比喻能一次性讲清它所有的设计。

工位 = Stage(阶段)

整车不是一个人从头造到尾,而是分成若干工位:起草设计 → 实现 → 检查 → 交付。每个工位职责单一。

专职工人 = sub-agent

每个工位站着一个专门的 AI 工人,只看自己需要的图纸(spec/plan),干完活就交一张工单回执,不和别人闲聊。

质检员 = 脚本门禁(Gate)

每个工位下线前必须过质检。质检不是工人自己说"我觉得行",而是跑一段脚本,不通过就不放行。

质检单 = receipt

每个工位都留下一张结构化回执(JSON)。所有回执串起来,就是这条工单完整、可审计的证据链。

关键洞察:单个 AI 容易"幻觉自己做完了"。b2r 的破解之道是——不相信任何 agent 的口头汇报,只相信脚本的退出码和文件里的证据。所有"是否通过"的判断,都由确定性的脚本做,而不是由 LLM 自评。这就是它能把"差不多得了"变成"真正 Done"的根本原因。

读懂原理 · 2

三大支柱:事实源、底盘、调度者

b2r 的整个系统建立在三个分工明确的层上。看懂这三层,就看懂了它的架构。

支柱一 · 唯一权威

事实源 state/*.md

state/ 里的 Markdown 文件是唯一权威:当前在做什么、队列里有哪些工单、路线图、客户可见变化……看板 BOARD.html 只是渲染产物,从不手改。冲突时一切以 state/* 为准。

支柱二 · 物理实现

底盘 workflow/scripts

一组 Node.js 脚本:校验状态、生成看板、推进工单、核验交付、检查红线……它们把"规则"变成可执行的代码。底盘只能由初始化命令一次性生成,不允许 agent 临时重造,避免版本漂移。

支柱三 · 编排大脑

调度者 the skill

就是 b2r skill 本身。它读取配置,按固定执行链路调度 sub-agent,在每步之间亲自跑脚本兜底,处理失败与升级。它是把"底盘"真正驱动起来的那个大脑。

为什么要分这三层?因为"规则"如果只写在 prompt 里,LLM 会慢慢忘、慢慢飘。把规则下沉成脚本(底盘),把状态固化成文件(事实源),LLM 只负责调度与创作(调度者)——这样即使换一个 AI 模型来跑,约束依然成立。这就是它"Agent 中立"的底气。

读懂原理 · 3

Agent 角色团:11 个各司其职的工人

每个角色都是一份 agents/<role>.md prompt 模板。调度者派工时把工单的具体信息填进占位符,启动一个全新上下文的 sub-agent。角色之间互不读对方的内部推理——比如评审员看不到起草者的思考过程,这样评审才是真正独立的"第二双眼睛"。

roadmap-planner

agents/roadmap-planner.md

把 roadmap 拆成带依赖的 Planned 工单,并给每条打复杂度标签(Stage 0/1)。

spec-drafter

agents/spec-drafter.md

把一条工单写成结构化 spec:做什么、不做什么、涉及哪些文件、验收命令。

plan-drafter

agents/plan-drafter.md

把 spec 拆成可执行的实施计划,按 TDD 步骤分切片。

spec-plan-reviewer

agents/spec-plan-reviewer.md

独立审 spec/plan:范围裁剪、不做项、文件清单是否合理(L2/L3 必跑)。

implementor

agents/implementor.md

先写跑红的测试,再写最小实现,让测试变绿。Stage 3 主力。

arch-security-reviewer

agents/arch-security-reviewer.md

架构 + 安全评审,命中红线即停(L3 必跑)。

direct-fix

agents/direct-fix.md

极简工单(typo/注释)专用:一个 agent 编辑+翻档+提交一气呵成。

handoff-committer

agents/handoff-committer.md

实现提交 + 状态翻档 + 交付提交,保证 commit 边界干净(Stage 5)。

ui-designer

agents/ui-designer.md

可选 UI 线:产出设计锚点与每屏 mockup。

design-reviewer

agents/design-reviewer.md

可选 UI 线:审设计锚点 / delta,给 PASS / NEEDS_FIX。

e2e-verifier

agents/e2e-verifier.md

可选 E2E 线:在里程碑/批次边界跑端到端验收旅程。

什么时候派 sub-agent,什么时候调度者亲自做?需要全新上下文的创作(起草 spec、TDD 实现)、需要独立判断的评审 → 派 sub-agent。而跑脚本、读状态、和你对话、起草升级决策 → 调度者自己做。sub-agent 永远不会再启动 sub-agent,保持单层调度,责任链清晰。

读懂原理 · 4

六阶段流水线:一条工单怎么走完

这是 b2r 的固定执行链路。每个阶段都有专职执行者和必须通过的脚本门禁(绿色标签)。

0

Triage 复杂度分诊 roadmap-planner

给每条工单打 L0–L3 标签,决定它走"完整 / 中等 / 简化 / 单工位"哪条路。判据写入 0-triage.json,只能升档不能降档。

1

Roadmap → Backlog roadmap-planner validate:state

把 roadmap 拆成 Planned 工单序列(含依赖)。真实工单号由调度者脚本化分配,不让 AI 手算。

2

Promote 推进为 Ready spec-drafter + plan-drafter + reviewer validate:state · deps:graph

把一条 Planned 翻成 Ready:生成 spec、plan、context-pack,并经独立评审。前置依赖必须全部 Done 才能 promote。

3

Implement 实现 implementor 红 gate · targeted · regression

工单翻 In Progress,TDD 红 → 绿:先有跑红的测试做证据,再写最小实现,最后定向测试 + 回归全过。

4

Review 架构与安全评审 arch-security-reviewer lint:redlines

L3 走完整评审,L2 走轻量内嵌,L1 跳过。红线 lint 命中即停,不带病往下走。

5

Commit & Handoff 交付 handoff-committer verify:handoff

实现提交(不含状态文件)+ 状态翻档 + 交付提交(只含状态文件)。两段物理分离,可独立回滚。最后核验交付完整性。

Stage 5 之后,工单回到 Idle,调度者亲跑 milestone:status 判断是否到了里程碑边界——没到就回 Stage 2 拿下一条;到了且配置了 E2E,就进入循环之外的里程碑端到端验收。整个过程不靠"感觉",全靠脚本报到边界。

读懂原理 · 5

四档复杂度:小事走快道,大事走全程

不是每条工单都要走完整六阶段。Stage 0 给工单分诊到 L0–L3,越简单的工单跳过越多环节,把精力留给真正复杂的改动。

档位判据(满足任一)走的路径跳过
L0 极简typo / 注释 / 文档措辞 / 单行格式化direct-fix 单 agent:改 + 翻档 + 提交spec / plan / review / arch 全跳过
L1 简单单文件 + 无新接口 + 无 schema + ≤30 行spec/plan 合并 + 自评 → 实现 → 交付独立 reviewer、arch-reviewer
L2 标准跨 2–3 文件 + 有新函数 + 无跨模块边界变化spec → plan → 独立 review → 实现 → 轻量评审 → 交付reviewer 走轻量清单,不调外部 skill
L3 复杂跨模块 / 新 schema / 新依赖 / 安全敏感 / 含 migration完整七工位全程—

只升不降:如果实现时发现改动其实比初判复杂(比如动了 schema),门禁会强制升档,重新走更严格的路径;但绝不允许把复杂工单偷偷降级成简单路径蒙混过关。

读懂原理 · 6

质检门禁与失败处理:出错了怎么办

b2r 把失败分成两类,区别对待——这是它既"自动化"又"不失控"的关键。

每个阶段的脚本门禁

调度者在每个 sub-agent 完成后,亲自跑脚本确认通过才进下一阶段。sub-agent 自己说"通过了"不算数。

阶段质检命令通过判据
Stage 1 Backlognpm run validate:state0 error
Stage 2 Promotevalidate:state + deps:graph0 error,依赖图无环无孤儿
Stage 3 红 gate核 3-impl.json有"测试先跑红"的证据
Stage 3 回归config 里的 regressionCommands全部退出码 0
Stage 4 Reviewnpm run lint:redlines0 命中
Stage 5 Handoffnpm run verify:handoff <id>全部检查通过

失败的三级处理

① 交付失败 → 自愈

sub-agent 返回了但产物不可用(报错串/空/截断)。这是基础设施问题,不惊动你:自动重派一次 → 仍不行调度者亲自接手。

② 质量失败 → 自动重试 ×1

产物合法但没过 gate。调度者把"哪条不通过 + 期望"打包,派一个全新 sub-agent 针对性修正,再过 gate。

③ Manager Override → 你拍板

重试仍失败,或 sub-agent 带证据自报阻塞。调度者渲染失败卷宗、起草建议,用选择题请你决策并留下决策记录。

绝不静默循环:重试失败必定升级到你面前,让你看到完整失败链——不会出现 AI "再试一次又一次"假装搞定的情况。Manager 的 5 个决策动作(接受 override / 降档 / 收缩范围 / 拆切片 / 放弃)各有明确回流点,全程留痕在 retro.md。

读懂原理 · 7

十条铁律:违反就当场停下

这些是从大量实战踩坑沉淀下来的硬约束。任何 sub-agent 在任何阶段命中其中一条,必须立即停下并报告,不允许"先记下、下轮再说"。它们是 b2r 不退化成"看心情写代码"的护栏。

#铁律它在防什么
1同时只有一条活跃工单(exactly one active)防 AI "假装在同时做好几件事"的幻觉
2事实源单一:state/* 权威,BOARD.html 只能脚本生成防状态与看板各说各话
3TDD 强制顺序:测试先跑红,才能写实现防"先写实现再补测试"的退化
4commit 物理分离 + 工作目录隔离让任一切片可独立回滚,防 cwd 泄漏到父仓
5前置 Done 才能 promote防把"想象的接口"当真接口去 spec
6架构红线触发即停不带病往下走
7自报阻塞必须带证据(grep/测试输出/条款)防空口"搞不动"式偷懒早退
8pipeline-status.json 由主线单写防 sub-agent 与主线并发写文件冲突
9skill 不直接写底盘脚本,只能 bootstrap 分发防 LLM 每次"贴心重造"导致版本漂移
10交付失败 ≠ 质量失败,前者自愈不惊动你区分基础设施抖动与真正的质量问题
读懂原理 · 8

两条可选支线:UI 设计线 与 E2E 验收线

这两条线默认关闭,只有在 workflow.config.mjs 里配置了 ui / e2e 块才启用。纯后端 / CLI / 库项目完全不受影响,也不会因为缺少 UI 或验收文件而报错。

UI 设计线

为带界面的工单补上"先设计、再实现"的环节,让实现有 mockup 可对照,而不是 AI 随手发挥。

  • 谁需要:有前端 / 界面改动的项目(配置 ui 块开启)。
  • 怎么触发:Stage 0 时 roadmap-planner 按文件路径给工单打 ui=true/false。
  • Stage 1.5 设计锚点:首个 UI 工单先由 ui-designer 生成 state/ui-anchor.md——项目级设计语言、共享外壳、导航与组件约定,经 design-reviewer 审过才作数。
  • Stage 2.0 每屏 mockup:每个 UI 工单再生成 work/<id>/ui/ 下的具体屏 mockup,同样过 reviewer。
  • 设计系统从哪来:优先用配置的 designRefs → 主动发现项目里的 theme/token/组件 → 都没有才用 ui-ux-pro-max 合成一套最小设计系统。
  • 怎么落地:mockup 写入 spec §4,implementor 把它们当成实现目标。

E2E 验收线

在工单一条条做完之后,站在用户视角跑端到端旅程,确认"拼起来真的能用",而不只是单元测试绿。

  • 谁需要:有可启动应用、需要按客户旅程验收的项目(配置 e2e 块开启)。
  • 两种粒度:里程碑级(一个里程碑全 Done 后验收)与批次级(一次提交 mint 出的一批工单作为验收单位),可并存。
  • 边界靠脚本判定:每次交付后跑 milestone:status / e2e-group:status,脚本报到边界才进 E2E,不靠 AI 凭感觉。
  • 怎么验:e2e-verifier 把"客户旅程"展开成真实验证流——探索段启动真应用取证,固化段把验过的旅程写成回归测试。
  • 两份产物:<milestone>-acceptance.md(业务语言报告,给人看)+ e2e-<milestone>.json(receipt,判 gate 用)。
  • 收口:PASS 不打扰你,作为里程碑翻档(Contract Done → Demo Ready)的前置证据;FAIL 自动按旅程去重生成修复工单,绝不静默循环。

事前 vs 事后两本账:acceptance.md 是事前验收基准(每个里程碑要验哪些客户旅程、用什么业务标准判 PASS/FAIL);customer-visible.md 是事后交付事实(每条 Done 工单实际交付了什么)。E2E 线就是拿"事前基准"去核对"事后事实"。

/skill-review 复盘回灌时
(人工触发)		 当 retro.md 的「待回灌清单」指出某个 agents/*.md 模板该改进时,在人监督下用它评审并修订模板。b2r 只负责让这笔"回灌债"可见,从不自动改模板。 可选 · 人工运维

一句话:纯后端 / CLI / 库项目跑完整流水线,一个外部 skill 都不用装。UI 与 E2E 是"按需挂载"的两条线;verifySkill 更是完全可换——你的项目用什么测试栈,就让它驱动什么。每个 stage 用了哪些 skill,都会记进该阶段 receipt 的 skills_used 字段,事后可审计。

动手使用

三步上手

几乎不用记命令——把链接和文档丢给你的 AI 助手,它自己跑。

安装 · 把 GitHub 链接丢给 agent

直接对你的 AI 助手说:

帮我安装这个 skill:https://github.com/edward-zyz/blueprint2real

agent 会自动把仓库 clone 到 skills 目录、装好底盘依赖并自测。装完开一个新会话即可识别。

初始化 · 第一次用自动建工作区

/blueprint2real

在你的项目里第一次调用时,skill 发现还没有 b2r-process/ 工作区,会引导并跑权威的 bootstrap 命令,一次性生成 state 文件、底盘脚本和配置。之后这一步不再重复。

接活 · 把产品文档 / 蓝图喂进去

/blueprint2real "你写好的产品文档 / 产品蓝图"

agent 接管整条流水线:拆工单 → 起草 spec/plan → TDD 实现 → 评审 → 干净交付,每一步脚本兜底。你通常只在 Manager Override 需要拍板时才被请来。

过程中你可能会说的几句话

你说agent 做什么
promote APP-001把这条工单推进到 Ready:起草 spec/plan 并独立评审
开始 APP-001翻 In Progress,走 TDD 红→绿实现
handoff APP-001评审 + 干净提交 + 核验交付,工单 Done
"看看进度 / 刷新看板"跑 render:board、validate:state,浏览器打开 BOARD.html

这些触发词都会唤起 b2r:开始 / promote / handoff / "跑工单流水线" / "从 roadmap 拆工单" / "让 agent 团跑完这条线",或在含 b2r-process/ 的仓库里要求"按工单走一遍"——不必每次念出 skill 名字。

融会贯通

一条工单的完整旅程

把前面所有概念串起来,看一条 L2 工单从无到 Done 发生了什么——你会看到"调度者 + sub-agent + 脚本 + 证据"是如何协同的。

1

你给 roadmap

调度者派 roadmap-planner 拆出 APP-007,分诊为 L2,写入 0-triage.json。跑 validate:state 通过。

2

promote APP-007

确认前置依赖全 Done → spec-drafter + plan-drafter 起草 → spec-plan-reviewer 独立审过。工单翻 Ready。

3

开始 APP-007

npm run start APP-007 原子翻 In Progress。implementor 先写跑红的测试(红 gate 证据落 3-impl.json),再写最小实现让它变绿,回归全过。

4

轻量评审

L2 走轻量内嵌评审 + lint:redlines 0 命中。

5

handoff APP-007

handoff-committer:实现 commit(不含 state)→ 翻档 → 交付 commit(只含 state + BOARD)。verify:handoff APP-007 全过。工单 Done。

留下证据链

work/APP-007_.../receipts/ 下躺着 0→5 每一阶段的 receipt JSON;commit 历史里实现与状态各自独立——这条工单从此可审计、可回滚。

常见问题

FAQ

Q:我必须用 Claude Code 吗?

不必。b2r 是 Agent 中立的——它只依赖 Markdown、JSON、Shell 和 Node.js。只要你的 agent runtime 支持 skill 机制并能调用 sub-agent 与跑脚本,就能用。

Q:纯后端 / CLI 项目能用吗?UI 和 E2E 是必须的吗?

能用。UI 设计线和 E2E 验收线都是可选的——只有在 workflow.config.mjs 里配置了 ui / e2e 块才启用。缺省即关闭,纯库/CLI 项目不会因此报错。

Q:旧项目用的是 dev/ 目录,还能兼容吗?

可以。默认工作目录是 b2r-process/(v5.1 起),但 skill 会按顺序回退查找 legacy 的 dev/ 路径,两者都支持。

Q:BOARD.html 能手动编辑吗?

不能。它是 render:board 的渲染产物,手改会被下次渲染覆盖。要改状态,请改 state/*.md 这个事实源。

Q:为什么不让 AI 自己一次性把所有工单都 promote?

因为铁律 #5:前置依赖全部 Done 才能 promote。否则后面的 spec 就是基于"想象中的接口"写的,必然失真。一次只推进一条,是为了让每条 spec 都基于真实已交付的代码。

Q:出了问题,AI 会不会自己瞎搞?

不会无限瞎搞。质量失败最多自动重试一次,仍不过就升级到 Manager Override——渲染完整失败链、起草建议,用选择题请你拍板,并把决策记录留痕在 retro.md。绝不静默循环。

Q:怎么做整体验证?

# 仓库级一键验证
npm run validate
# 或分别跑
python3 .github/scripts/quick_validate.py .
cd bootstrap/workflow && npm test
bash evals/run-all-assertions.sh
